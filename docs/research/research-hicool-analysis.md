## 1. 赛题深度解析

### 1.1 赛题原文摘要

**赛题名称**：HICOOL 大赛题目 - MTClaw 终极形态：桌面级"快准狠"全能 Agent 工作站

**赛题背景**：MTClaw 已在编程场景验证了 Function Router 架构的优雅与高效，但 MTT AIBOOK 的用户不只有程序员——他们要查资料、写周报、读论文、做数据、还得偶尔和 AI 闲聊放松。每一项都召唤完整的 openclaw 链路既笨重又浪费本地算力。

**使命**：基于 MTClaw 进行二次开发，打造跑在 MTT AIBOOK 本地、覆盖全工作场景的"快准狠"个人 AI 工作站。用户通过自然语言输入，Router 自动把请求派发给最合适的专家 Subagent。

### 1.2 核心任务要求

| 要求项 | 具体内容 | Prometheus 对应 |
|--------|---------|----------------|
| 必选技术栈 | 基座框架：MTClaw | ✅ 基于 MTClaw Function Router 二开 |
| 交付环境 | MTT AIBOOK (AIOS 1.4.2) 本地运行演示 | ✅ 部署在 AIBOOK，使用本地路由模型 |
| Subagent 数量 | 至少新增 3 个高价值垂直 Subagent | ✅ 新增 5 个（RAG/记忆/写作/日程/闲聊） |
| 自动分发 | 由 Function Router 自动分发，无需手动 @ 或切换 | ✅ temperature=0.0 确定性路由 + 优先级排序 |
| 场景痛点 | 解决真实可产品化的场景痛点 | ✅ 5 个痛点场景（文档检索/偏好记忆/写作/日程/闲聊） |
| 资源匹配 | 轻量任务不走重型链路 | ✅ 闲聊路由模型直回（1-3s vs 15-40s） |
| 演示要求 | 连续对话涵盖 4+ 领域，全程自动路由，无人工干预 | ✅ 10 轮演示剧本覆盖 5 个领域 + 通用兜底 |
| 可安装使用 | 可直接打包安装使用 | ✅ 一键 install.sh + 预置 Subagent 市场 |
| 源代码提交 | 巅峰赛 40 名选手提交源代码 | ✅ 代码合入 MTClaw 仓库 |

### 1.3 赛题参考方向分析

赛题提供了 9 个参考 Subagent 方向：

| 方向 | 场景价值 | Prometheus 选择 | 选择理由 |
|------|---------|----------------|---------|

| 办公助手 | 邮件、钉钉等办公软件支持 | ❌ 未选 | 依赖第三方 API，演示稳定性不可控 |

### 1.5 赛题核心命题洞察

通过对赛题的深度分析，提炼出以下核心命题：

1. **"快准狠"是评分主线**：三个字分别对应三个评分维度，是评委的核心评判框架

## 2. 评分维度拆解与对标策略

### 2.1 评分维度全景

赛题设置了 4 个主评分维度 + 3 个加分项（不设上限）：

```
评分体系
├── 1. 快（速度）
│   ├── Router 决策延迟
│   ├── 首 token 延迟
│   └── 端到端响应耗时与 openclaw 原生链路的对比加速比
│
├── 2. 准（准确率）
│   ├── Router 分发准确率
│   ├── 各 Subagent 任务完成质量与领域专业度
│   └── 通用 Benchmark 智商不变低
│
├── 3. 狠（深度）
│   ├── 场景覆盖广度（Subagent 数量与类型多样性）
│   └── 产品化完成度（是否可直接交付/安装/使用）
│
├── 4. 生态和商业价值
│   ├── 真实场景价值
│   ├── 产品化潜力
│   ├── 市场与商业模式
│   ├── 生态适配价值
│   ├── 可复用与可扩展性
│   └── 示范带动作用
│
└── 加分项（不设上限）
    ├── Router 自学习
    ├── 可视化路由追踪
    └── 开箱即用
```

### 2.2 维度一：快——速度优势

#### 评分要求拆解

| 评分指标 | 赛题要求 | Prometheus 目标 | 数据类型 | 技术支撑 |
|---------|---------|----------------|---------|---------|
| Router 决策延迟 | 越低越好 | < 1s | [目标] | temperature=0.0 确定性路由，路由模型轻量 |
| 首 token 延迟 | 越低越好 | < 2s (L1) / < 5s (L2) | [目标] | 分层延迟策略，闲聊路由模型直回 |
| 端到端响应耗时 | 越低越好 | < 3s (L1) / < 10s (L2) / < 40s (L3) | [推测] | 三层延迟分层 |
| 加速比 vs openclaw | 越高越好 | 4.99x~6.85x | [推测] | MTClaw 团队 50 任务 benchmark 数据参考 |

#### 分层延迟策略

```
L1 即时层（~25% 请求）
  ├── 记忆 Subagent: < 1s (SQLite 本地查询)
  └── 闲聊 Subagent: 1-3s (路由模型直回，跳过上游 LLM)

L2 快速层（~35% 请求）
  ├── RAG Subagent: 1-3s (ChromaDB 混合检索)
  └── 日程 Subagent: < 1s (SQLite 本地查询)

L3 标准层（~25% 请求）
  └── 写作 Subagent: 10-40s (上游 LLM 生成)

兜底层（~15% 请求）
  └── 通用推理: 15-40s (上游 LLM 全量推理)
```

#### 加速比论证

MTClaw 团队在 50 个系统控制任务上的实测数据（非 Prometheus 实测）：

| 模式 | Pass@1 | 平均耗时 | 加速比 |
|------|--------|----------|--------|
| Baseline（纯上游 LLM） | 99.0% | 37.97s | 1.00x |
| Permissive | 95.5% | 5.54s | **6.85x** |
| Strict | **100.0%** | 7.61s | **4.99x** |

**重要说明**：上述数据在 7 个系统控制工具场景下测得。Prometheus 使用 16 个工具，场景复杂度更高，实际加速比预计会降低。Prometheus 自身实测数据需在方案验证阶段产出。

#### "快"的竞争策略

| 策略 | 实现方式 | 演示效果 |
|------|---------|---------|
| 闲聊秒回 | 路由模型直回，不走上游 LLM | "讲个笑话" 1-3s vs 传统 15-30s |
| 本地优先 | 4/5 Subagent 完全无网络依赖 | 日程查询 < 1s，偏好记忆 < 1s |
| Completion Check 快路径 | 70% 请求跳过上游 LLM [目标] | 工具结果直接格式化返回 |
| 分治架构 | 轻量任务走轻量链路 | 避免所有请求走同一条慢路径 |

### 2.3 维度二：准——准确率优势

#### 评分要求拆解

| 评分指标 | 赛题要求 | Prometheus 目标 | 数据类型 | 技术支撑 |
|---------|---------|----------------|---------|---------|
| Router 分发准确率（初始） | 越高越好 | > 90% | [目标] | 5 Subagent + 优先级 + 防误判 + temperature=0.0 |
| Router 分发准确率（自学习后） | 越高越好 | > 95% | [目标] | Router 自学习引擎持续优化 |
| 双层路由误判率 | 越低越好 | < 8% | [目标] | L2 确认路由零误判 + L1 高置信度低误判 |
| 各 Subagent 任务完成质量 | 100% 可演示 | 100% 可演示 | [目标] | 每个 Subagent 独立测试 |
| 通用 Benchmark 智商不变低 | 不退化 | 不退化 | 设计保证 | 100% 透明转发上游 LLM |

#### 准确率保障机制（五层防护）

```
第 1 层：Subagent 专职化 + 工具数控制
  → 16 个工具（接近 LLM function calling 最佳范围 <15）
  → 每个工具描述 < 200 字，互不重叠

第 2 层：temperature=0.0 确定性路由
  → 同样的输入永远选择同一个 Subagent
  → 零随机性，可复现

第 3 层：双重意图匹配 + 优先级排序
  → LLM function calling (主) + 关键词/正则匹配 (辅)
  → 优先级：记忆(P1) > 写作(P2) > 日程(P3) > RAG(P4) > 闲聊(P5) > 兜底(P6)

第 4 层：防误判保护（保守策略）
  → 宁可假阴性（走上游 LLM），不可假阳性（走 chat_light）
  → complex 消息绝对禁止路由到闲聊 Subagent

第 5 层：路由 fallback 安全网
  → 路由模型超时/不可用 → 直接转发上游 LLM
  → 最差情况退化为直接使用上游 LLM（智商不降，只是慢）
```

#### "智商不变低"论证

赛题要求"通用 Benchmark 智商不变低"，这是对 FR 架构的关键质疑点。Prometheus 的保障：

```
未命中任何 Subagent → 100% 透明转发上游 LLM
  ├── prompt 不做任何改写
  ├── 响应不做任何截断
  └── 原样流式透传

性能增益 = Subagent 命中率 × 各 Subagent 的定制优化
智商兜底 = 100% 的上游 LLM 能力（零衰减）
```

**论证逻辑**：Prometheus 只在用户意图明确匹配某 Subagent 时介入（加速+专业化），其余请求原样转发上游 LLM。上游 LLM 的能力是 100% 保留的，不存在"智商降低"问题。

#### 加分项深度分析

**Router 自学习——最核心的差异化加分项**

赛题原文的关键词是"动态优化**分发策略**"。这意味着：

| 层次 | 含义 | 是否满足赛题 |
|------|------|------------|
| 偏好记忆注入 | 用户说"以后用 Markdown" → 写入 memory → 注入生成 prompt | ❌ 只影响"内容生成偏好"，不影响路由 |
| 路由策略优化 | 用户修正路由 → 调整关键词权重/提示词/优先级 → 下次自动正确路由 | ✅ 真正"动态优化分发策略" |

Prometheus v3.1 的 Router 自学习引擎实现了第二层，通过四个机制真正优化路由分发：

1. **路由置信度评分**：从路由模型 logprob 计算置信度，量化路由决策把握程度
2. **双层路由**：低置信度时主动询问用户（L2 确认路由），避免误路由
3. **用户修正反馈**：记录 4 种修正场景（L2 确认后选择 / L1 路由后纠正 / 用户主动澄清 / 用户弃权）
4. **路由策略动态调整**：四种调整机制（关键词权重 / 路由提示词片段 / Subagent 优先级 / 置信度阈值校准）

### 3.3 架构同类竞品分析

#### MTClaw Function Router vs 其他 Agent 路由架构

| 维度 | MTClaw FR | LangChain Router | AutoGen | CrewAI |
|------|-----------|-----------------|---------|--------|
| 路由方式 | LLM function calling (temperature=0) | LLM 路由链 | 对话式协商 | 角色任务分配 |
| 接入方式 | OpenAI-compatible 零侵入 | SDK 集成 | SDK 集成 | SDK 集成 |
| 工具执行 | Shell wrapper (stdin JSON → stdout JSON) | Python 函数 | Python 函数 | Python 函数 |
| 上游兜底 | 透明代理上游 LLM | 无 | 无 | 无 |
| 延迟优化 | 小模型路由 + 大模型兜底 | 无 | 无 | 无 |
| 生态绑定 | OpenClaw / AIBOOK | LangChain 生态 | Microsoft 生态 | 独立 |

**MTClaw FR 的核心优势**：
1. **零侵入接入**：任何能配置 `base_url` 的客户端都能无缝接入
2. **延迟分层**：小模型路由 + 大模型兜底，实现加速
3. **工具委托**：将工具执行委托回 Agent 框架，避免"幽灵工具调用"
4. **Completion Check**：工具执行后判断是否需要上游 LLM 补充

**Prometheus 在 MTClaw 基础上的增量**：
1. **5 个垂类 Subagent**：从系统控制扩展到办公/学习/日程/社交
2. **即时偏好引擎**：跨会话记忆 + 偏好自动注入
3. **Router 自学习引擎**：置信度评分 + 双层路由 + 策略动态调整
4. **Subagent 市场机制**：CLI + registry + 安装/卸载/更新
5. **路由追踪面板**：可视化路由决策路径

#### 第二幕：Router 自学习进化演示（5 轮，~2 分钟）

```
[第 1 轮] 用户: "帮我看看 GPU 算力对比"
  系统: 路由模型置信度 0.42 (< 0.45)
        → L2 确认路由: "你是想[查文档]还是[闲聊]? (1/2)"
  用户: "1" (查文档)
  系统: 路由到 RAG → 检索 → 返回结果
  记录: 修正 (chat_light → rag_search), 置信度 0.42
  → 展示：低置信度时主动询问，避免误路由

[第 2 轮] 用户: "帮我看看 模型对比"
  系统: 路由模型置信度 0.58 (中间区, 走 top-1)
        → 路由到 chat_light (top-1 仍是 chat_light)
  用户: "不对，我要查文档"
  系统: 识别纠正意图 → 重新路由到 RAG
  记录: 修正 (chat_light → rag_search), 置信度 0.58
  触发: 关键词权重调整 ("帮我看看" + 技术名词 → RAG)
  → 展示：用户修正反馈 + 策略调整触发

[第 3-4 轮] (类似输入，系统逐步学习)
  → 展示：路由提示词动态增强

  系统: 路由提示词已注入 "用户历史: '帮我看看' + 技术名词 → RAG"
        路由模型置信度 0.82 (≥ 0.75)
        → L1 自动路由到 RAG
  用户零额外输入
  → 展示：根据用户使用习惯动态优化分发策略
```

**演示要点**：
- 第 1 轮：展示双层路由（L2 确认），建立用户信任
- 第 2 轮：展示用户修正反馈和策略调整触发
