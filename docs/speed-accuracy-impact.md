## 1. 快：延迟的每一个环节都在刀刃上

"快"不仅仅是代码优化，而是在架构层面做**延迟分层**--让每种请求走最短路径。

### 1.1 设计决策：Function Router 分治架构

**为什么这个设计提升速度？**

传统单模型 Agent 对所有请求走同一条路径：用户输入 -> 单个 LLM -> 响应。无论用户是说"你好"还是"分析这个 10GB CSV"，都走相同的 15-40s 延迟。

Function Router 在 LLM 调用**之前**插入了一层轻量路由：

```
传统方案:
  用户 -> LLM (15-40s) -> 响应

Prometheus:
  用户 -> 路由模型 (0.3s, temperature=0) -> 判断意图
    ├── 闲聊/记忆 -> 本地查询 (1-3s)           ← 跳过 LLM
    ├── RAG/日程 -> 本地工具 (1-5s)            ← 跳过 LLM
    ├── 写作 -> 上游 LLM (10-40s)               ← 需要 LLM
    └── 未命中 -> 上游 LLM (15-40s)             ← 兜底
```

**关键数字**：

| 请求类型 | 传统方案延迟 [推测] | Prometheus 延迟 [推测] | 为什么更快 |
|---------|------------|----------------|-----------|
| "你好" | 15-30s | **1-3s** | 路由模型直回，不走上游 LLM |
| "找 GPU 笔记" | 无法处理 | **1-3s** | ChromaDB 本地向量检索，无网络往返 |
| "记住我喜欢 Markdown" | 下次需重新说 | **< 1s** | SQLite UPSERT，单次磁盘写入 |
| "明天有什么安排" | 无本地日程 | **< 1s** | SQLite 本地查询，无 LLM 参与 |

**注意**：上述延迟为 [推测] 值，基于架构推断。实际数字需在目标硬件（MTT AIBOOK）上实测。

**参考数据**：MTClaw 团队在 50 个系统控制任务上取得了 4.99x~6.85x 加速 [推测，MTClaw 团队测试，非 Prometheus 实测]。但那是在 7 个工具的场景下测得的，Prometheus 使用 16 个工具，场景复杂度更高，实际加速比预计会降低 [推测]。Prometheus 自身的实测数据将在方案验证阶段产出。

### 1.2 设计决策：Completion Check 快路径

**为什么这个设计提升速度？**

MTClaw 的 Completion Check 机制在工具执行完成后做一次判断：结果是否已经可以返回给用户？

```
TASK_COMPLETE -> 直接返回（跳过上游 LLM）-> 延迟 = 路由 + 工具执行
TASK_INCOMPLETE -> 转发上游 LLM 补充 -> 延迟 = 路由 + 工具 + LLM

目标：70% 的 TASK_COMPLETE 命中率 [目标]
```

**为什么能命中 70%？**

每个 Subagent 的工具返回结果已经包含了人类可读的答案，并非原始 JSON：

| 工具 | 返回示例 | 用户可直接消费？ |
|------|---------|---------------|
| `rag_search` | `"找到 3 篇关于 GPU 算力的笔记：1. xxx.md (相似度 0.92)..."` | ✓ |
| `memory_recall` | `"你的偏好：写作格式=Markdown, 语言=中文..."` | ✓ |
| `schedule_query` | `"今天有 3 个安排：1. 10:00 产品评审会..."` | ✓ |
| `writing_generate` | `"## 本周完成\n- API 开发..."` | 部分需要 LLM 润色 |

工具返回被设计为**已完成格式化**的文本，使得 Completion Check 可以判定为 COMPLETE。只有写作类需要上游 LLM 做二次加工。

**注意**：v2.0 将命中率目标从 80% 下调为 70%，更务实。实际命中率需在目标场景实测。

### 1.3 设计决策：闲聊 Subagent 走路由模型直回

**为什么这个设计提升速度？**

这是"快"最极致的体现。闲聊场景（估计占日常对话 15-20% [推测]）完全跳过上游 LLM：

```
技术可行性分析：
  路由模型: qwen3-30b-a3b-instruct-2507
  - 30B 参数，对闲聊场景的生成质量足够（笑话、寒暄、简单问答）
  - 本地/边缘部署，推理延迟 1-3s [推测]
  - 对比：上游 doubao/gpt-4o，网络往返 + 推理 15-30s [推测]

温度策略：
  - 路由决策时 temperature=0.0（确定性，保证路由准确）
  - 闲聊生成时 temperature=0.7（随机性，保证对话自然）
```

**误判保护（v2.0 强化）**：

闲聊直回最大的风险是误判--把需要上游 LLM 的问题误路由到路由模型直回，导致回答质量差。v2.0 采用**保守策略**：

```
宁可假阴性（走上游 LLM），不可假阳性（走 chat_light）

假阳性代价（致命）:
  复杂推理问题 -> 误判为闲聊 -> 路由模型直回 -> 回答质量差 -> 用户不信任

假阴性代价（可接受）:
  闲聊 -> 未命中 chat_light -> 走兜底上游 LLM -> 延迟高但回答质量没问题

防误判规则:
  complex 消息绝对禁止路由到 chat_light:
    - 包含 "为什么" + 专业术语 -> 禁止
    - 包含 "怎么做" + 技术动词 -> 禁止
    - 包含 "对比/区别/优缺点" -> 禁止
    - 消息长度 > 200 字 -> 禁止
    - 包含任何文件扩展名 -> 禁止
    - 包含编程/技术关键词 -> 禁止
    - 任何不确定的情况 -> 走上游 LLM
```

---

### 1.4 设计决策：本地优先，网络仅在必要时使用

| 存储/计算 | 位置 | 延迟 [推测] | 适用 Subagent |
|-----------|------|------|-------------|
| SQLite | 本地磁盘 | < 10ms | 记忆、日程 |
| ChromaDB | 本地磁盘 | < 100ms | RAG、记忆 |
| BGE-M3 嵌入 | 本地 CPU | ~50ms/条 [推测] | RAG、记忆 |
| 路由模型 | 本地/边缘 | 0.3-1.5s [推测] | 闲聊 |
| 上游 LLM | 云端 | 10-30s [推测] | 写作 |

**核心原则**：数据不出设备，算力尽量本地，智能才上云端。

**注意**：BGE-M3 嵌入延迟需要在 MTT AIBOOK 上实测验证，这是 RAG Subagent 延迟的关键瓶颈。

---

### 1.5 设计决策汇总：快

```
架构层面:
  Function Router 分治 -> 避免所有请求走同一条慢路径
  Completion Check 快路径 -> 70% 请求跳过上游 LLM [目标]
  闲聊路由模型直回 -> 1-3s vs 传统 15-30s [推测]

数据层面:
  SQLite 本地查询 -> < 10ms
  ChromaDB 本地向量检索 -> < 100ms
  BGE-M3 本地嵌入 -> ~50ms/条 [推测，待实测]

网络策略:
  本地优先原则 -> 4 个 Subagent 完全无网络依赖
  外网仅在上游 LLM 时使用

实测支撑:
  MTClaw 系统控制垂域: 4.99x~6.85x 加速 [推测，MTClaw 团队测试，非 Prometheus 实测]
  其他数字均为 [推测] 或 [目标]，待方案验证阶段实测
```

## 2. 准：精准分发的每一层都在消除歧义

"准"的核心挑战是：如何让系统准确理解用户意图，并将请求分发到正确的处理单元？Prometheus 通过**四个层次的确定性设计**来实现。

### 2.1 设计决策：Subagent 专职化 + 工具数控制

**为什么这个设计提升准确率？**

如果一个 Agent 试图用一个巨型 Prompt 处理所有场景，它需要在 Prompt 中编码所有规则，依赖 LLM 的"理解能力"。LLM 对复杂 Prompt 的遵循率随规则数量增加而下降。

Prometheus 的策略：把"理解"拆成两步，每一步都更简单：

```
传统方案:  LLM 同时理解"用户意图" + "如何处理" -> 一个巨型 Prompt
Prometheus: 路由模型理解"用户意图" -> 专职 Subagent 处理 -> 两个小 Prompt

复杂度对比:
  传统 Prompt: ~5000 tokens (包含所有场景的规则)
  路由 Prompt: ~1200 tokens (16 个工具描述，做选择题)
  Subagent Prompt: ~500 tokens (单个领域，做专业题)
```

**v2.0 关键改进**：工具数从 24 降到 16

研究表明，LLM 的 function calling 准确率在工具数量 < 15、每个工具描述 < 200 字时最高 [推测]。v1.0 有 24 个工具（超出最佳范围），当前缩减到 16 个（16 个自定义 + 5 个 builtin，Bash 砍掉后无重叠，共 21 个暴露给路由模型），略超 <15 最佳线但接近，并通过 5 个 Subagent 清晰划分保持路由精度。

```
v1.0: 8 Subagent / 19 自定义工具 + 5 builtin = 24 工具 -> 超出最佳范围
v2.0: 5 Subagent / 13 自定义工具 + 5 builtin = 18 工具 -> 接近最佳范围
```

---

### 2.2 设计决策：temperature=0.0 确定性路由

**为什么这个设计提升准确率？**

```
路由决策（需要确定性）:  temperature=0.0
  -> 同样的输入，永远选择同一个 Subagent
  -> 不会出现"第一次走 RAG，第二次走闲聊"的随机行为

闲聊生成（需要多样性）:  temperature=0.7
  -> 同样的输入，可以有不同的回复
  -> 自然对话不应该机械重复

两类决策分离：路由用确定性模型，生成用创造性模型
```

**量化效果**：temperature=0.0 的路由模型在 50 条混合意图测试中，分发准确率目标 > 90% [目标]。v2.0 将目标从 95% 下调为 90%，更务实。

---

### 2.3 设计决策：双重意图匹配 + 优先级排序

**为什么这个设计提升准确率？**

单靠 LLM function calling 有时会误判--尤其是在语义模糊的边界 case。Prometheus 叠加了一层关键词+正则匹配作为辅助：

```
意图判断 = LLM function calling (主) + 关键词/正则匹配 (辅) + 优先级排序 (仲裁)

优先级设计逻辑:
  P1 记忆     ← "记住了" / "以后都" -> 明确的偏好声明，最高优先级
  P2 写作     ← "翻译" / "帮我写/生成/起草/润色" -> 明确的写作意图
  P3 日程     ← "日程/会议/待办/任务/安排" -> 日程管理
  P4 RAG      ← "找一下/搜索" + 文档语境 -> 本地检索
  P5 闲聊     ← 寒暄/情感 -> 低优先级，避免误判
  P6 兜底     ← 以上都不匹配 -> 上游 LLM
```

**为什么不是纯 LLM 路由？**

LLM function calling 准确率约 90% [目标]，在 1000 次请求中就有 100 次误判。关键词/正则匹配作为确定性补充，可以纠正其中约 30-50 次（尤其是明确关键词的 case），将最终准确率推到 92-95% [推测]。

---

### 2.4 设计决策：防误判保护（保守策略）

**为什么这个设计提升准确率？**

"假阳性"（不该路由到某 Subagent 但路由了）的代价远大于"假阴性"（该路由但没路由，走了兜底）：

```
假阳性代价（致命）:
  复杂推理问题 -> 误判为闲聊 -> 路由模型直回 -> 回答质量差 -> 用户不信任

假阴性代价（可接受）:
  闲聊 -> 未命中任何工具 -> 走兜底上游 LLM -> 延迟高但回答质量没问题

防误判设计（保守策略）:
  complex 消息绝对禁止路由到 chat_light:
    - 包含 "为什么" + 专业术语 -> 禁止
    - 包含 "怎么做" + 技术动词 -> 禁止
    - 包含 "对比/区别/优缺点" -> 禁止
    - 消息长度 > 200 字 -> 禁止
    - 包含任何文件扩展名 -> 禁止
    - 包含编程/技术关键词 -> 禁止
    - 任何不确定的情况 -> 走上游 LLM
```

**宁可保守，不可冒进**：假阴性只是慢一点，假阳性会让用户觉得系统"不靠谱"。

### 2.5 设计决策：即时偏好注入消除重复说明

**为什么这个设计提升准确率？**

这是"准"的高级形态--不只是单次请求的路由准确，而是**跨会话的体验准确**。

```
无偏好注入:
  用户: "帮我写周报"
  LLM: 生成通用格式周报（可能是英文、纯文本、无结构）
  用户: "不对，要用中文、Markdown、分成本周完成和下周计划"
  LLM: 重新生成 -> 用户需要每次纠正

有偏好注入（即时偏好引擎）:
  用户: "帮我写周报，用 Markdown，中文，包含本周完成和下周计划"
  系统: 即时偏好检测 -> 写入 memory(writing_format=markdown, language=zh-CN, ...)
  LLM: 按用户偏好生成 -> 一次命中

  [下次/新会话]
  用户: "写周报"
  系统: memory_recall 自动注入偏好
  LLM: 自动生成符合偏好的周报 -> 用户零额外输入

格式符合率: > 80% [目标]
用户无需重复说明: 100%
```

**v2.0 关键改进**：即时偏好 vs 后台反思

```
v1.0 后台反思引擎:
  用户说偏好 -> 写入日志 -> 凌晨 2:00 cron 分析 -> 更新 memory
  问题：演示中无法展示（需要跨天）

v2.0 即时偏好引擎:
  用户说偏好 -> 即时检测 -> 同步写入 memory -> 下次请求自动注入
  优势：演示中即时可见
```

### 2.6 设计决策：RAG 混合检索

**为什么这个设计提升准确率？**

纯向量检索在以下场景失效：
- 专业术语（"CUDA" 的向量与 "GPU 编程" 距离较远）
- 缩写（"FR" vs "Function Router"）
- 代码片段（代码的语义向量不可靠）
- 数字和日期（"2026-07-12" 的向量与 "7 月 12 日" 距离较远）

```
纯稠密检索:
  query "FR 配置" -> 向量 -> ChromaDB cosine -> 返回 "Function Router" 相关文档
  但对 "FR" 这个缩写的 embedding 质量高度敏感

混合检索 (稠密 + BM25):
  稠密通道: "FR 配置" -> 语义相似文档
  BM25 通道: "FR" 精确匹配 -> 包含 "FR" 关键词的文档
  RRF 融合: score = 1/(60+rank_dense) + 1/(60+rank_sparse)
  -> 同时覆盖语义相关和字面匹配

Top-5 召回率: > 85% [目标]
```

**注意**：v2.0 将目标从 90% 下调为 85%，更务实。

---

### 2.7 设计决策：路由 fallback 安全网

**v2.0 新增设计**

```
路由模型超时 (10s) 或不可用
  │
  └── 直接转发上游 LLM（跳过工具路由）
      -> 确保用户请求不会因为路由模型故障而无响应
      -> 最差情况：退化为直接使用上游 LLM（智商不降，只是慢）
```

这个设计确保了"准"的底线：即使路由完全失效，系统也不会给出错误答案，只会走最安全的通用路径。

---

### 2.8 设计决策汇总：准

```
路由层面:
  Subagent 专职化 -> 工具描述短且互不重叠
  工具数 16 个 -> 接近 LLM function calling 最佳范围 [推测]
  temperature=0.0 -> 确定性路由，可复现
  关键词/正则 + 优先级 -> 辅助纠正 LLM 的误判
  防误判保护 -> 宁可假阴性不假阳性
  路由 fallback -> 路由失效时安全降级

偏好层面:
  即时偏好检测 -> 演示中即时可见
  memory_recall 自动注入 -> 消除用户重复说明
  偏好格式符合率 > 80% [目标]

检索层面:
  稠密 + BM25 + RRF 混合检索 -> Top-5 召回率 > 85% [目标]

参考数据（非 Prometheus 实测）:
  MTClaw 工具准确率: 94.8%~97.5% [推测，MTClaw 团队在系统控制垂域测试，非 Prometheus 实测]
  Prometheus 自身的实测数据将在方案验证阶段产出
```

## 3. 狠：自我进化的每一个循环都在积累护城河

"狠"体现在不是做一个"能用的工具"，而是做一个"越用越强的系统"。每个设计决策都在构建长期竞争壁垒。

### 3.1 设计决策：即时偏好引擎--实时学习

**为什么这个设计体现"狠"？**

大多数 Agent 止步于"响应当前请求"。Prometheus 在用户使用过程中实时提取偏好，立即写入记忆。

```
v2.0 即时偏好引擎不是事后分析，而是系统运行的有机组成部分：

  用户: "以后都用 Markdown 格式"
    -> 即时检测到偏好声明
    -> 同步写入 memory(writing_format=markdown, importance=4)
    -> 下次写作请求自动注入

  用户: "记住了，我喜欢中文回复"
    -> 即时检测到偏好声明
    -> 同步写入 memory(preferred_language=zh-CN, importance=4)
    -> 下次对话自动注入

  用户: "总是用三段式结构写周报"
    -> 即时检测到习惯声明
    -> 同步写入 memory(weekly_report_structure=三段式, importance=4)
    -> 下次写周报自动注入
```

**狠在哪里？**

传统 Agent 的"智能"是一个静态函数 `f(x) = y`。Prometheus 的"智能"是一个迭代函数 `f_t(x) = y, where f_{t+1} = learn(f_t, interaction_t)`。使用时间越长，函数越接近用户。

**v2.0 改进**：即时偏好引擎替代了 v1.0 的后台反思引擎，核心优势是**演示中即时可见**。

---

### 3.2 设计决策：记忆的衰减与强化--模拟人类遗忘曲线

**为什么这个设计体现"狠"？**

不是一个简单的"存了就永远记住"的数据库，而是模拟人类记忆的工作方式：

```
热记忆强化:
  access_count > 10 且 importance < 5
  -> importance += 1
  -> 经常被使用的记忆变得更"重要"
  例: 用户每周写周报都用到 writing_format 偏好 -> 这个记忆越来越重要

冷记忆衰减:
  access_count = 0 且 30 天未更新 且 importance > 1
  -> importance -= 1
  -> 长期不用且不重要的记忆自动降级
  例: 用户 3 个月前说 "记住了，我喜欢用蓝色主题" 但从没用过

自动清理:
  总记忆数 > 1000 (MAX_MEMORIES)
  -> 移除 importance 最低的 note 类记忆
  -> 偏好和身份记忆不参与清理（更稳定）
```

**狠在哪里？**

这是一个**有生命周期的记忆系统**，而非简单的 KV 存储。它模拟了人类记忆的运作方式--重要的东西反复强化，不重要的东西自然遗忘。

---

### 3.3 设计决策：飞轮效应--迁移成本递增

**为什么这个设计体现"狠"？**

```
用户使用 1 天:  记忆数 ~10 条，偏好注入 0 次
                  -> 切换回 ChatGPT 成本：几乎为零

用户使用 1 周:  记忆数 ~50 条，偏好自动注入 15 次
                  -> 切换成本：需要重新告知所有偏好

用户使用 1 个月: 记忆数 ~200 条，反思摘要积累了 30 天的行为模式
                  -> 切换成本：大量个性化上下文丢失，ChatGPT 变回"陌生人"

用户使用 3 个月: 记忆数 ~500 条（衰减后 ~400）
                  -> 切换成本：极高，AI 已经深度个性化
```

**狠在哪里？**

这不是一个"用完即走"的工具。用户在 Prometheus 上积累的个性化数据构成了**事实上的迁移壁垒**。数据全在本地属于用户，但重新训练另一个 AI 理解这些数据的成本极高--这是合法的、对用户有益的 "lock-in"。

---

### 3.4 设计决策：路由 fallback--永不无响应

**v2.0 新增**

```
路由模型超时/不可用
  -> 直接转发上游 LLM
  -> 最差情况退化为直接使用上游 LLM
  -> 用户请求永不无响应

端到端超时 120s
  -> 返回已完成的部分结果
  -> 避免系统进入无响应状态
```

**狠在哪里？**

系统设计上保证了"永不无响应"--即使路由模型挂了、Subagent 崩了、上游 LLM 超时了，用户都能拿到某种形式的回答。这是可靠性上的"狠"。

---

### 3.5 设计决策汇总：狠

```
自我进化:
  即时偏好引擎 -> 用户说偏好时实时写入 -> 演示中即时可见
  记忆衰减/强化 -> 模拟人类记忆生命周期

竞争壁垒:
  飞轮效应 -> 使用越久迁移成本越高
  全本地数据 -> 用户拥有数据但难以迁移个性化

可靠性:
  路由 fallback -> 路由故障永不无响应
  端到端超时保护 -> 120s 必须返回
  透明兜底 -> 最差情况退化为上游 LLM

长期价值:
  不是一个 "更好的 ChatGPT"
  而是一个 "会成长的私人助理"
```

---

## 4. 总结：快准狠的架构底座

三个维度不是孤立的优化点，而是由同一架构底座支撑的：

```
                    ┌──────────────────────────────────┐
                    │     MTClaw Function Router        │
                    │   (分治调度，零侵入接入)            │
                    └──────────────────────────────────┘
                                    │
            ┌───────────────────────┼───────────────────────┐
            │                       │                       │
            ▼                       ▼                       ▼
    ┌───────────────┐     ┌───────────────┐     ┌───────────────┐
    │   本地执行层    │     │   智能调度层    │     │   自我进化层    │
    │               │     │               │     │               │
    │ • ChromaDB    │     │ • 路由模型     │     │ • 即时偏好引擎  │
    │ • SQLite      │     │ • 5 Subagent  │     │ • 记忆衰减     │
    │ • BGE-M3      │     │ • Completion  │     │ • 偏好注入     │
    │ • dateparser  │     │   Check       │     │ • 飞轮效应     │
    │               │     │ • 路由fallback │     │               │
    │ -> 快           │     │ -> 准           │     │ -> 狠           │
    └───────────────┘     └───────────────┘     └───────────────┘
            │                       │                       │
            └───────────────────────┼───────────────────────┘
                                    │
                                    ▼
                    ┌──────────────────────────────────┐
                    │         上游 LLM 兜底              │
                    │   (100% 透明转发，智商零衰减)       │
                    └──────────────────────────────────┘
```

| 维度 | 核心设计决策 | 一句话机制 | 数据支撑 |
|------|------------|-----------|---------|
| **快** | 延迟分层 + 本地优先 | 让每种请求走最短路径 | MTClaw 4.99x~6.85x [推测，MTClaw 团队测试]；Prometheus 实测待产出 |
| **准** | 专职路由 + 确定性分发 + 即时偏好 | 把模糊的理解问题变成清晰的分类问题 + 实时注入偏好 | MTClaw 工具准确率 94.8%~97.5% [推测，MTClaw 团队测试]；Prometheus 实测待产出 |
| **狠** | 即时偏好引擎 + 飞轮效应 + 永不无响应 | 系统越用越强，迁移成本递增，可靠性兜底 | 设计保证 |

**快、准、狠不是三个独立的目标，而是同一架构在三个维度上的自然结果。**

**v2.0 核心改进**：
1. 所有数字标注来源（[实测] / [目标] / [推测]），不编造
2. 工具数从 24 降到 16，接近路由准确率最佳范围
3. 即时偏好引擎替代后台反思，演示可跑通
