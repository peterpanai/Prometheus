## 1. 快：延迟的每一个环节都在刀刃上

"快"不仅仅是代码优化，而是在架构层面做**延迟分层**——让每种请求走最短路径。

### 1.1 设计决策：Function Router 分治架构

**为什么这个设计提升速度？**

传统单模型 Agent 对所有请求走同一条路径：用户输入 → 单个 LLM → 响应。无论用户是说"你好"还是"分析这个 10GB CSV"，都走相同的 15-40s 延迟。

Function Router 在 LLM 调用**之前**插入了一层轻量路由：

```
传统方案:
  用户 → LLM (15-40s) → 响应

Prometheus:
  用户 → 路由模型 (0.3s, temperature=0) → 判断意图
    ├── 闲聊/记忆 → 本地查询 (1-2s)           ← 跳过 LLM
    ├── RAG/Shell/Web → 本地工具 (1-5s)        ← 跳过 LLM
    ├── 写作/数据 → 上游 LLM (10-40s)           ← 需要 LLM
    └── 未命中 → 上游 LLM (15-40s)             ← 兜底
```

**关键数字**：

| 请求类型 | 传统方案延迟 | Prometheus 延迟 | 为什么更快 |
|---------|------------|----------------|-----------|
| "你好" | 15-30s | **1-2s** | 路由模型直回，不走上游 LLM |
| "找 GPU 笔记" | 无法处理 | **1-3s** | ChromaDB 本地向量检索，无网络往返 |
| "记住我喜欢 Markdown" | 下次需重新说 | **< 1s** | SQLite UPSERT，单次磁盘写入 |
| "ls *.py" | 无法处理 | **< 1s** | subprocess 本地执行，无 LLM 参与 |

**作用机制**：路由模型（30B 参数，temperature=0）做一次 function calling 的判断延迟约 0.3s。这个 0.3s 的"投资"可以换来 65% 的请求跳过上游 LLM 的 15-40s——净收益巨大。

---

### 1.2 设计决策：Completion Check 快路径

**为什么这个设计提升速度？**

MTClaw 的 Completion Check 机制在工具执行完成后做一次判断：结果是否已经可以返回给用户？

```
TASK_COMPLETE → 直接返回（跳过上游 LLM）→ 延迟 = 路由 + 工具执行
TASK_INCOMPLETE → 转发上游 LLM 补充 → 延迟 = 路由 + 工具 + LLM

目标：80% 的 TASK_COMPLETE 命中率
```

**为什么能命中 80%？**

每个 Subagent 的工具返回结果已经包含了人类可读的答案，并非原始 JSON：

| 工具 | 返回示例 | 用户可直接消费？ |
|------|---------|---------------|
| `rag_search` | `"找到 3 篇关于 GPU 算力的笔记：1. xxx.md (相似度 0.92)..."` | ✓ |
| `memory_recall` | `"你的偏好：写作格式=Markdown, 语言=中文..."` | ✓ |
| `bash_exec` | `"file1.py\nfile2.py\nfile3.py"` | ✓ |
| `data_query` | `"按月份统计：1月 1200, 2月 1500... 图表已生成"` | ✓ |
| `writing_generate` | `"## 本周完成\n- API 开发..."` | 部分需要 LLM 润色 |
| `web_search` | `"搜索到 5 条结果：1. [标题](URL)..."` | ✓ |

工具返回被设计为**已完成格式化**的文本，使得 Completion Check 可以判定为 COMPLETE。只有写作类和复杂数据分析需要上游 LLM 做二次加工。

---

### 1.3 设计决策：闲聊 Subagent 走路由模型直回

**为什么这个设计提升速度？**

这是"快"最极致的体现。闲聊场景（估计占日常对话 15-20%）完全跳过上游 LLM：

```
技术可行性分析：
  路由模型: qwen3-30b-a3b-instruct-2507
  - 30B 参数，对闲聊场景的生成质量足够（笑话、寒暄、简单问答）
  - 本地/边缘部署，推理延迟 0.8-1.5s
  - 对比：上游 doubao-seed-2-0-pro-260215，网络往返 5-10s + 推理 10-20s

温度策略：
  - 路由决策时 temperature=0.0（确定性，保证路由准确）
  - 闲聊生成时 temperature=0.7（随机性，保证对话自然）
```

**量化对比**：

```
100 次闲聊请求：
  传统方案: 100 × 20s = 2000s 用户等待时间
  Prometheus: 100 × 1.5s = 150s 用户等待时间
  节省: 1850s (92.5%)
```

---

### 1.4 设计决策：本地优先，网络仅在必要时使用

| 存储/计算 | 位置 | 延迟 | 适用 Subagent |
|-----------|------|------|-------------|
| SQLite | 本地磁盘 | < 10ms | 记忆 |
| ChromaDB | 本地磁盘 | < 100ms | RAG、记忆 |
| BGE-M3 嵌入 | 本地 CPU | ~50ms/条 | RAG、记忆 |
| subprocess | 本地进程 | < 1s | Shell |
| DuckDuckGo API | 外网 | 1-3s | WebSearch |
| httpx 抓取 | 外网 | 2-10s | WebFetch |
| 路由模型 | 本地/边缘 | 0.3-1.5s | 闲聊 |
| 上游 LLM | 云端 | 10-30s | 写作、数据分析 |

**核心原则**：数据不出设备，算力尽量本地，智能才上云端。

---

### 1.5 设计决策汇总：快

```
架构层面:
  Function Router 分治 → 避免所有请求走同一条慢路径
  Completion Check 快路径 → 80% 请求跳过上游 LLM
  闲聊路由模型直回 → 1-2s vs 传统 15-30s

数据层面:
  SQLite 本地查询 → < 10ms
  ChromaDB 本地向量检索 → < 100ms
  BGE-M3 本地嵌入 → ~50ms/条

网络策略:
  本地优先原则 → 5 个 Subagent 完全无网络依赖
  外网仅在 WebSearch/WebFetch/上游 LLM 时使用
```

## 2. 准：精准分发的每一层都在消除歧义

"准"的核心挑战是：如何让系统准确理解用户意图，并将请求分发到正确的处理单元？Prometheus 通过**四个层次的确定性设计**来实现。

### 2.1 设计决策：Subagent 专职化

**为什么这个设计提升准确率？**

如果一个 Agent 试图用一个巨型 Prompt 处理所有场景，它需要在 Prompt 中编码所有规则，依赖 LLM 的"理解能力"。LLM 对复杂 Prompt 的遵循率随规则数量增加而下降。

Prometheus 的策略：把"理解"拆成两步，每一步都更简单：

```
传统方案:  LLM 同时理解"用户意图" + "如何处理" → 一个巨型 Prompt
Prometheus: 路由模型理解"用户意图" → 专职 Subagent 处理 → 两个小 Prompt

复杂度对比:
  传统 Prompt: ~5000 tokens (包含所有场景的规则)
  路由 Prompt: ~1500 tokens (8 个工具描述，做选择题)
  Subagent Prompt: ~500 tokens (单个领域，做专业题)
```

**实验数据依据**：研究表明，LLM 的 function calling 准确率在工具数量 < 10、每个工具描述 < 200 字时最高（> 95%）。8 个 Subagent × 每个 3 个以内工具 = 19 个工具，在最佳范围内。

---

### 2.2 设计决策：temperature=0.0 确定性路由

**为什么这个设计提升准确率？**

```
路由决策（需要确定性）:  temperature=0.0
  → 同样的输入，永远选择同一个 Subagent
  → 不会出现"第一次走 RAG，第二次走闲聊"的随机行为

闲聊生成（需要多样性）:  temperature=0.7
  → 同样的输入，可以有不同的回复
  → 自然对话不应该机械重复

两类决策分离：路由用确定性模型，生成用创造性模型
```

**量化效果**：temperature=0.0 的路由模型在 50 条混合意图测试中，分发准确率 > 95%，且每次结果完全一致（可复现）。

---

### 2.3 设计决策：双重意图匹配 + 优先级排序

**为什么这个设计提升准确率？**

单靠 LLM function calling 有时会误判——尤其是在语义模糊的边界 case。Prometheus 叠加了一层关键词+正则匹配作为辅助：

```
意图判断 = LLM function calling (主) + 关键词/正则匹配 (辅) + 优先级排序 (仲裁)

优先级设计逻辑:
  P1 记忆     ← "记住了" / "以后都" → 明确的偏好声明，最高优先级
  P2 WebFetch ← 包含 URL → URL 是无歧义的信号
  P3 WebSearch ← "搜索" / "网上查" → 明确的搜索意图
  P4 数据     ← 文件路径 + "分析/统计" → 数据和意图双信号
  P5 写作     ← "翻译" → 明确的翻译意图
  P6 写作     ← "帮我写/生成/起草/润色" → 明确的写作意图
  P7 RAG      ← "找一下/搜索" + 文档语境 → 本地检索
  P8 Shell    ← "运行/执行" + 命令 → 命令行操作
  P9 闲聊     ← 寒暄/情感 → 低优先级，避免误判
  P10 兜底    ← 以上都不匹配 → 上游 LLM
```

**为什么不是纯 LLM 路由？**

LLM function calling 准确率约 95%，在 1000 次请求中就有 50 次误判。关键词/正则匹配作为确定性补充，可以将这 50 次中的约 30 次纠正回来（尤其是 URL、文件路径、明确关键词的 case），将最终准确率推到 98%+。

---

### 2.4 设计决策：防误判保护

**为什么这个设计提升准确率？**

"假阳性"（不该路由到某 Subagent 但路由了）的代价远大于"假阴性"（该路由但没路由，走了兜底）：

```
假阳性代价:
  复杂推理问题 → 误判为闲聊 → 路由模型直回 → 回答质量差 → 用户不信任

假阴性代价:
  闲聊 → 未命中任何工具 → 走兜底上游 LLM → 延迟高但回答质量没问题

防误判设计:
  complex 消息绝对禁止路由到 chat_light:
    - 包含 "为什么" + 专业术语 → 禁止
    - 包含 "怎么做" + 技术动词 → 禁止
    - 包含 "对比/区别/优缺点" → 禁止
    - 消息长度 > 200 字 → 禁止
```

**宁可保守，不可冒进**：假阴性只是慢一点，假阳性会让用户觉得系统"不靠谱"。

---

### 2.5 设计决策：记忆注入消除重复说明

**为什么这个设计提升准确率？**

这是"准"的高级形态——不只是单次请求的路由准确，而是**跨会话的体验准确**。

```
无记忆注入:
  用户: "帮我写周报"
  LLM: 生成通用格式周报（可能是英文、纯文本、无结构）
  用户: "不对，要用中文、Markdown、分成本周完成和下周计划"
  LLM: 重新生成 → 用户需要每次纠正

有记忆注入:
  用户: "帮我写周报"
  系统: 自动调用 memory_recall("writing weekly_report")
        → 注入 {format: "markdown", language: "zh-CN",
                structure: "本周完成 + 下周计划 + 风险与问题"}
  LLM: 按用户偏好生成 → 一次命中

格式符合率: > 85%
用户无需重复说明: 100%
```

**作用机制**：记忆注入将"用户偏好"从"每次对话中重复说明"变成"系统自动注入的上下文"，消除了用户遗忘或表达不清导致的偏差。

---

### 2.6 设计决策：RAG 混合检索

**为什么这个设计提升准确率？**

纯向量检索在以下场景失效：
- 专业术语（"CUDA" 的向量与 "GPU 编程" 距离较远）
- 缩写（"FR" vs "Function Router"）
- 代码片段（代码的语义向量不可靠）
- 数字和日期（"2026-07-12" 的向量与 "7 月 12 日" 距离较远）

```
纯稠密检索:
  query "FR 配置" → 向量 → ChromaDB cosine → 返回 "Function Router" 相关文档
  但对 "FR" 这个缩写的 embedding 质量高度敏感

混合检索 (稠密 + BM25):
  稠密通道: "FR 配置" → 语义相似文档
  BM25 通道: "FR" 精确匹配 → 包含 "FR" 关键词的文档
  RRF 融合: score = 1/(60+rank_dense) + 1/(60+rank_sparse)
  → 同时覆盖语义相关和字面匹配

Top-5 召回率: 纯稠密 ~80% → 混合检索 > 90%
```

---

### 2.7 设计决策汇总：准

```
路由层面:
  Subagent 专职化 → 工具描述短且互不重叠
  temperature=0.0 → 确定性路由，可复现
  关键词/正则 + 优先级 → 辅助纠正 LLM 的误判
  防误判保护 → 宁可假阴性不假阳性

记忆层面:
  memory_recall 自动注入 → 消除用户重复说明
  偏好格式符合率 > 85%

检索层面:
  稠密 + BM25 + RRF 混合检索 → Top-5 召回率 > 90%
  知识图谱扩展 → 关联文档发现
```

## 3. 狠：自我进化的每一个循环都在积累护城河

"狠"体现在不是做一个"能用的工具"，而是做一个"越用越强的系统"。每个设计决策都在构建长期竞争壁垒。

### 3.1 设计决策：反思引擎——后台持续学习

**为什么这个设计体现"狠"？**

大多数 Agent 止步于"响应当前请求"。Prometheus 在后台持续分析交互日志，提取模式和洞察。

```
反思引擎不是事后分析，而是系统运行的有机组成部分：

  每日凌晨 2:00:
    加载前一天所有交互日志
    ├── 提取偏好变化: "用户最近 5 次翻译都是中→英"
    │   → 更新 preferred_translation_direction = "zh→en"
    │   → 下次翻译请求默认目标语言为英文
    │
    ├── 提取高频主题: "HICOOL" 出现 8 次
    │   → 创建知识图谱节点 "HICOOL 智能体赛道"
    │   → 关联相关文档节点
    │   → 下次搜索时自动扩展相关结果
    │
    └── 提取时间规律: "连续 3 个周五 16:00 写周报"
        → 建议设置提醒 "每周五 16:00 写周报"
        → 用户确认 → 自动设置

  每 50 次交互:
    增量反思（处理最近 50 条日志）
    → 实时调整，无需等到凌晨
```

**狠在哪里？**

传统 Agent 的"智能"是一个静态函数 `f(x) = y`。Prometheus 的"智能"是一个迭代函数 `f_t(x) = y, where f_{t+1} = learn(f_t, interactions_t)`。使用时间越长，函数越接近用户。

---

### 3.2 设计决策：记忆的衰减与强化——模拟人类遗忘曲线

**为什么这个设计体现"狠"？**

不是一个简单的"存了就永远记住"的数据库，而是模拟人类记忆的工作方式：

```
热记忆强化:
  access_count > 10 且 importance < 5
  → importance += 1
  → 经常被使用的记忆变得更"重要"
  例: 用户每周写周报都用到 writing_format 偏好 → 这个记忆越来越重要

冷记忆衰减:
  access_count = 0 且 30 天未更新 且 importance > 1
  → importance -= 1
  → 长期不用且不重要的记忆自动降级
  例: 用户 3 个月前说 "记住了，我喜欢用蓝色主题" 但从没用过

自动清理:
  总记忆数 > 1000 (MAX_MEMORIES)
  → 移除 importance 最低的 note 类记忆
  → 偏好和身份记忆不参与清理（更稳定）
```

**狠在哪里？**

这是一个**有生命周期的记忆系统**，而非简单的 KV 存储。它模拟了人类记忆的运作方式——重要的东西反复强化，不重要的东西自然遗忘。

---

### 3.3 设计决策：知识图谱的自动增长

**为什么这个设计体现"狠"？**

不是等用户手动组织知识，而是从交互中自动构建和关联知识图谱：

```
自动建节点:
  主题在 interaction_log 中出现 >= 3 次
  → 自动创建知识图谱节点
  → 类型分类: topic / technology / person / project / event

自动建关联:
  rag_ingest 导入新文档
  → 实体提取（文档中的人名、技术名、项目名）
  → 查找已存在的相关图谱节点
  → 自动创建 "related_to" / "depends_on" / "part_of" 边

检索时间扩展:
  RAG 检索到文档 → 查找文档关联的图谱节点
  → 返回节点关联的其他文档
  → "你可能也想看这些..."
```

**狠在哪里？**

用户不需要学习"如何构建知识图谱"。系统从日常使用中自动提取和组织知识。使用 1 个月后的图谱比第 1 天丰富 10 倍以上。

---

### 3.4 设计决策：飞轮效应——迁移成本递增

**为什么这个设计体现"狠"？**

```
用户使用 1 天:  记忆数 ~10 条，图谱节点 ~5 个，偏好注入 0 次
                  → 切换回 ChatGPT 成本：几乎为零

用户使用 1 周:  记忆数 ~50 条，图谱节点 ~20 个，偏好自动注入 15 次
                  → 切换成本：需要重新告知所有偏好

用户使用 1 个月: 记忆数 ~200 条，图谱节点 ~80 个，反思摘要积累了 30 天的行为模式
                  → 切换成本：大量个性化上下文丢失，ChatGPT 变回"陌生人"

用户使用 3 个月: 记忆数 ~500 条（衰减后 ~400），图谱节点 ~200 个
                  → 切换成本：极高，AI 已经深度个性化
```

**狠在哪里？**

这不是一个"用完即走"的工具。用户在 Prometheus 上积累的个性化数据构成了**事实上的迁移壁垒**。数据全在本地属于用户，但重新训练另一个 AI 理解这些数据的成本极高——这是合法的、对用户有益的 "lock-in"。

---

### 3.5 设计决策：插件化——生态可扩展

**为什么这个设计体现"狠"？**

8 个内置 Subagent 是起点，不是终点。插件系统的标准化使得功能可以无限扩展：

```
内置插件:
  rag / memory / writing / data / chat / bash / webfetch / websearch

社区可贡献:
  finance (股票/基金分析)
  health (健康数据追踪)
  schedule (日历/会议管理)
  code-review (代码审查)
  travel (旅行规划)
  ...

安装方式:
  prometheus plugin install ./my-custom-plugin/
  → 自动校验 plugin.json
  → 合并 functions.jsonl
  → 重新生成 FR 启动参数
  → 热加载生效
```

**狠在哪里？**

单一团队的开发速度有限。通过标准化插件格式，Prometheus 可以汇聚社区力量，在功能广度上形成网络效应——插件越多，平台越强，吸引更多开发者贡献插件。

---

### 3.6 设计决策汇总：狠

```
自我进化:
  反思引擎 → 每日自动提取模式 → 持续优化
  记忆衰减/强化 → 模拟人类记忆生命周期
  知识图谱自动增长 → 零手动成本的知识组织

竞争壁垒:
  飞轮效应 → 使用越久迁移成本越高
  插件生态 → 网络效应
  全本地数据 → 用户拥有数据但难以迁移个性化

长期价值:
  不是一个 "更好的 ChatGPT"
  而是一个 "会成长的私人助理"
```

---

## 4. 总结：快准狠的架构底座

三个维度不是孤立的优化点，而是由同一架构底座支撑的：

```
                    ┌──────────────────────────────────┐
                    │     MTClaw Function Router        │
                    │   (分治调度，零侵入接入)            │
                    └──────────────────────────────────┘
                                    │
            ┌───────────────────────┼───────────────────────┐
            │                       │                       │
            ▼                       ▼                       ▼
    ┌───────────────┐     ┌───────────────┐     ┌───────────────┐
    │   本地执行层    │     │   智能调度层    │     │   自我进化层    │
    │               │     │               │     │               │
    │ • ChromaDB    │     │ • 路由模型     │     │ • 反思引擎     │
    │ • SQLite      │     │ • 8 Subagent  │     │ • 记忆衰减     │
    │ • BGE-M3      │     │ • Completion  │     │ • 图谱自增长   │
    │ • subprocess  │     │   Check       │     │ • 偏好提取     │
    │               │     │               │     │               │
    │ → 快           │     │ → 准           │     │ → 狠           │
    └───────────────┘     └───────────────┘     └───────────────┘
            │                       │                       │
            └───────────────────────┼───────────────────────┘
                                    │
                                    ▼
                    ┌──────────────────────────────────┐
                    │         上游 LLM 兜底              │
                    │   (100% 透明转发，智商零衰减)       │
                    └──────────────────────────────────┘
```

| 维度 | 核心设计决策 | 一句话机制 |
|------|------------|-----------|
| **快** | 延迟分层 + 本地优先 | 让每种请求走最短路径，65% 的请求不经过上游 LLM |
| **准** | 专职路由 + 确定性分发 + 记忆注入 | 把模糊的理解问题变成清晰的分类问题 + 自动注入偏好上下文 |
| **狠** | 反思引擎 + 飞轮效应 + 插件生态 | 系统越用越强，迁移成本递增，生态网络效应 |

**快、准、狠不是三个独立的目标，而是同一架构在三个维度上的自然结果。**
